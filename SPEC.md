# SPEC — Asker (бот «Герман»)

**Статус:** 🟡 Planning
**Владелец:** bklv-kirill
**Создан:** 2026-04-24

## Описание

Телеграм-бот под кодовым именем **«Герман»**. Конкретный функционал бота будет определён в отдельной фазе — сейчас проект находится на стадии подготовки инфраструктуры разработки.

Имя приложения — `Asker` (используется как префикс в docker-compose и в `APP_NAME` из `.env`).

## Архитектура (текущее состояние)

Один сервис:

| Слой | Технология |
|---|---|
| Runtime | Go 1.25 |
| Упаковка | Docker (образ на базе `golang:1.25-alpine`) |
| Оркестрация | docker-compose (один bridge-network `app`) |
| Dev-режим | Hot reload через `air` (`github.com/air-verse/air`) |
| Секреты | `.env` в корне (шаблон — `.env.example`) |
| Конфиг | `github.com/spf13/viper` — загрузка `.env` + env vars в единую структуру `Config` |
| Telegram | `github.com/go-telegram/bot` — клиент Bot API в режиме long-polling |
| Хранилище | SQLite через `github.com/mattn/go-sqlite3` (cgo) + голый `database/sql`; файл в bind-mount `./data/` → `/data/` в контейнере |
| Логирование | `log/slog` (stdlib) — root-логгер создаётся в `main`, передаётся компонентам DI-стилем |

Внешние зависимости (Redis, сторонние API) **пока отсутствуют**. Появятся после решения о функционале бота.

## Как запустить локально

```bash
cp .env.example .env
# заполнить TOKEN_BOT_TOKEN (получить у @BotFather) — без валидного токена приложение упадёт
docker compose up --build
```

После старта в stdout идёт лог вида `time=... level=INFO msg=starting app=Asker bot=Герман` (формат `slog.TextHandler`), затем бот ждёт апдейты в long-polling. На команду `/start` от пользователя отвечает `Привет, <FirstName>! Я <BotName>.` (новый пользователь) или `Рад тебя снова видеть, <FirstName>!` (известный). На любое другое текстовое сообщение срабатывает echo-хендлер: бот повторяет текст дословно. Каждое входящее событие (`command_in`/`message_in`) и каждое успешное исходящее (`message_out`) пишется в журнал `telegram_events` (см. таблицу) — в stdout идёт сжатый INFO-лог `telegram_events created` с `id`/`event`/`telegram_user_id`. При изменении `.go` файлов `air` автоматически пересобирает и перезапускает бинарник.

## Переменные окружения

Загружаются через `internal/config` (viper): сначала читается `.env` в корне проекта (если есть), затем поверх накладываются переменные окружения процесса — env vars имеют приоритет. Публичный API: `config.Load() *Config` (при ошибке — `panic`, дальнейшее выполнение без конфига не имеет смысла).

Все поля — **обязательные**: при пустом значении `config.Load` паникует с сообщением `config: <KEY> is required but empty`.

| Имя | Поле `Config` | Назначение | Обязательная |
|---|---|---|---|
| `APP_NAME` | `AppName` | Префикс имени контейнера и идентификатор приложения | да |
| `BOT_NAME` | `BotName` | Отображаемое имя бота (используется в приветствии `/start`) | да |
| `TOKEN_BOT_TOKEN` | `TokenBotToken` | Токен Telegram Bot API | да |
| `DB_PATH` | `DBPath` | Путь к SQLite-файлу внутри контейнера (dev: `/data/asker.db`) | да |

Дополнительно пробрасываются стандартные HTTP-proxy переменные — они нужны только Go-рантайму (`http.DefaultTransport` подхватывает их автоматически) и в `config.Config` не попадают. Требуются на хостах, где `api.telegram.org` заблокирован (например, российский прод-сервер): без прокси `bot.New` падает на `getMe` с `context deadline exceeded`. Если оставить пустыми — Go пойдёт напрямую.

| Имя | Назначение |
|---|---|
| `HTTP_PROXY` | HTTP-прокси для исходящих запросов |
| `HTTPS_PROXY` | HTTPS-прокси для исходящих запросов (используется для `api.telegram.org`) |
| `NO_PROXY` | Список хостов в обход прокси |

## Фазы реализации

- [x] **Фаза 0 — Скаффолд инфраструктуры.** docker-compose + Dockerfile + air + базовый Go-луп `working...`.
- [x] **Фаза 1 — Подключение к Telegram (БАЗА).** Клиент `github.com/go-telegram/bot` в long-polling, структура `TelegramBot` с `NewTelegramBot(token, botName, logger, users, telegramUsers, telegramEvents)` и `Start(ctx)`, graceful shutdown по SIGINT/SIGTERM. `/start` различает нового и возвратившегося пользователя (проверка через `telegramUsers.GetByTelegramUserID`): новый — сохраняется в `telegram_users` и получает «Привет, <FirstName>! Я <BotName>.», старый без привязанного номера — «Рад тебя снова видеть, <FirstName>!» + предложение привязать; старый с привязанным номером — короткое «Рад тебя снова видеть, <FirstName>!». Default-хендлер (echo) перед ответом тоже вызывает `CreateNewTelegramUserIfNotExists` — страховка на случай общения без `/start`. Привязка номера: inline-кнопка «Привязать номер» → callback_query → reply-keyboard с request_contact → contact_in → users.Create + telegramUsers.SetUserIDByTelegramUserID. Primary signal Фазы 1 (start + echo) подтверждён ранее; flow привязки номера — следующее, что нужно проверить вживую. Расширение набора команд — в следующих фазах.
- [ ] **Фаза 2 — Функционал бота «Герман».** Определяется отдельно.
- [~] **Фаза 3 — Хранилище.** SQLite подключён: пакет `internal/storage/sqlite` с `New(cfg, logger) *sql.DB` (open + ping, panic при ошибке — консистентно с `config.Load`), файл лежит в bind-mount `./data/asker.db`. Добавлены миграции `0001_users` (доменный пользователь: name/gender/age/info/phone — `phone TEXT NOT NULL UNIQUE` с CHECK «только цифры, не пустая»), `0002_telegram_users` (запись о Telegram-аккаунте: `telegram_user_id UNIQUE`, `first_name NOT NULL`, `last_name`/`username` nullable, `user_id INTEGER UNIQUE REFERENCES users(id) ON DELETE SET NULL` — опциональная привязка к доменному `users`, NULL до момента, когда пользователь привязал номер телефона; триггер `AFTER UPDATE` на `updated_at`) и `0003_telegram_events` (журнал TG-событий: `telegram_user_id` FK на `telegram_users(id)` с `ON DELETE CASCADE`, `payload TEXT NOT NULL CHECK (json_valid(payload))`, `created_at` секундной гранулярности; индекс `(telegram_user_id, created_at DESC)`). Миграции лежат в `migrations/` как раздельные `.up.sql` / `.down.sql`; скрипт `scripts/refresh_db.sh` делает полный refresh (все down в обратном порядке → все up в прямом, с остановкой/стартом контейнера). **Автоматический runner миграций при старте приложения намеренно не делаем** — настройка БД лежит на человеке: перед первым `docker compose up` и каждый раз, когда нужно обнулить схему, запускается `./scripts/refresh_db.sh`. Репозитории: `usersRepo` (`Create(ctx, name, phone)`), `telegramUsersRepo` (`Create`/`ExistsByTelegramUserID`/`GetByTelegramUserID`), `telegramEventsRepo` (`Create` — payload `json.RawMessage`); доменные модели — в `internal/models`. Подключение `telegramEventsRepo` в `TelegramBot` и запись событий в хендлерах — следующий шаг.
- [ ] **Фаза 4 — Prod-деплой.** Multi-stage Dockerfile, systemd / compose-стек на сервере, nginx (если нужен webhook).

Path в логе оставлен как `cfg.DBPath` (чистый путь), чтобы оператор не путался. Проверено на runtime: 5 последовательных `PRAGMA foreign_keys` через `*sql.DB` возвращают `1`.
- **2026-04-25** — Фаза 3 (telegram_users.user_id): в миграцию `0002_telegram_users.up.sql` добавлена колонка `user_id INTEGER UNIQUE REFERENCES users(id) ON DELETE SET NULL` — опциональная FK-привязка TG-аккаунта к доменному `users`. NULL до момента, когда пользователь привязал номер и для него появилась запись `users` (тогда `user_id` проставляется отдельным UPDATE-методом — добавим, когда придёт сценарий). UNIQUE: один доменный пользователь = один TG-аккаунт; SQLite допускает множество NULL в UNIQUE-колонке, поэтому до привязки строки не конфликтуют. ON DELETE SET NULL: удаление users (например, по запросу на забвение) сбрасывает связь, но сам telegram_users остаётся — журнал событий не теряется. **Edit-in-place** в применённой миграции по аналогии с предыдущим случаем (`0001_users.phone`); обоснование то же — dev-фаза, refresh_db.sh пересоздаёт схему с нуля. **Замечание:** `PRAGMA foreign_keys = ON` сейчас в проекте не включается (ни в DSN, ни после `Open` в `internal/storage/sqlite/sqlite.go`), поэтому FK на уровне БД не enforce'ятся в рантайме — это касается и существующего FK `telegram_events.telegram_user_id → telegram_users(id)`.
